# Authenticated X Mint Radar Cron Pattern

Use this pattern when the user asks to monitor X/Twitter for NFT mint signals such as `free mint`, `allowlist`, `WL`, `stealth mint`, `mint live`, or `FCFS`, especially for Ethereum NFT research.

## When to use

- User wants ongoing monitoring instead of a one-off search.
- Public web search is too stale/noisy and user provides X cookies (`auth_token`, `ct0`) for an authenticated X session.
- Target delivery is a Telegram group/topic, e.g. `telegram:<chat_id>:<thread_id>`.

## Secret handling

- Store X cookies in a private env file such as `~/.hermes/x-radar.env` with `0600` permissions.
- Never print cookie values back into chat or logs.
- Output only public tweet metadata: tweet ID/URL, public text, keyword hits, rough score, and red flags.
- Keep the watcher read-only. Do not like/repost/follow/DM or submit any wallet/on-chain action from the radar.

Example env shape:

```bash
X_AUTH_TOKEN=<auth_token>
X_CT0=<ct0>
```

## Authenticated X GraphQL search

X public web search can work for a baseline scan, but authenticated GraphQL is more timely. Plain `urllib` requests may return `404` even when cookies are valid. Use the same request pattern as `x-tg-notify`:

1. `curl_cffi.Session(impersonate="chrome136")`
2. Cookies: `auth_token`, `ct0`, `lang=en` on `.x.com`
3. Headers:
   - `Authorization: Bearer <public X web bearer>`
   - `X-Csrf-Token: <ct0>`
   - `X-Twitter-Auth-Type: OAuth2Session`
   - `X-Twitter-Active-User: yes`
   - `X-Client-Transaction-Id` generated by `x-client-transaction-id`
4. Call `https://x.com/i/api/graphql/<SearchTimelineQID>/SearchTimeline` with variables `{rawQuery,count,querySource:"typed_query",product:"Latest"}`.

The `SearchTimeline` query ID rotates. Scrape it from X `main.*.js` when needed by searching for `operationName:"SearchTimeline"` and the nearby `queryId:"..."`.

Known working deps:

```bash
python3 -m pip install 'curl_cffi>=0.7.0' 'x-client-transaction-id>=0.0.1' 'beautifulsoup4>=4.12.0'
```

## Filtering and scoring

Useful triggers:

- `free mint`
- `allowlist`, `WL`
- `stealth mint`, `stealth launch`
- `mint live`
- `FCFS`
- `0x` + `NFT`/`ERC721`

Filter out:

- generic giveaway bait with no contract/site/timing
- token/airdrop posts that are not NFT mints
- drainer/phishing wording (`seed phrase`, `private key`, suspicious claim urgency)
- stale posts unless still actionable

Prefer candidates with:

- contract address or official mint site
- explicit supply, price, date/time, chain
- credible official/project account or known marketplace
- Ethereum/mainnet context

## Telegram delivery

If the target is a forum topic, use the thread-aware target:

```text
telegram:<chat_id>:<thread_id>
```

For no-agent cron jobs, make the script print the exact final Telegram message. Empty stdout means no delivery, so print a short "no qualifying signals" message if the user expects heartbeat updates.

## Cron shape

Use a script-only cron for low-cost polling:

```text
schedule: every 15m
script: x_nft_mint_radar_format.py
no_agent: true
deliver: telegram:<chat_id>:<thread_id>
```

The formatter should run the authenticated search script, dedupe via a local state file, and print concise Indonesian output:

- Heading: `🚨 X NFT Mint Radar — authenticated scan`
- Up to 5 bullets/items
- Each item: score, trigger keywords, source URL, one-line summary, risk note, next step (`verify official/site/contract, dry-run before execute`)

## Safety reminder

This radar only discovers leads. Before minting anything, still run the normal `nft-auto-mint` workflow: verify official source, inspect contract/dApp, check supply/price/permissions, simulate/dry-run, and require explicit user confirmation before any transaction.
